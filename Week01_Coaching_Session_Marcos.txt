AI COACH: Good question! We've covered the core of it, but let me round it out a bit more so you have the full picture of Express routing before we move on.

A few more pieces that make up "Express routing" as a whole:

- HTTP methods: app.get(), app.post(), app.put(), app.delete(), etc. — each corresponds to a different type of request, like we just discussed.
- Route parameters: dynamic segments in a URL, like /categories/:id, where :id becomes accessible as req.params.id.
- Query strings: extra data in the URL like /categories?sort=name, accessible via req.query.
- Middleware: functions that run before your final route handler — useful for things like logging, authentication, or parsing data. They can be applied to a single route or globally with app.use().
- Route order matters: Express checks routes top-to-bottom and uses the first one that matches.
- Routers: for bigger apps, you can split routes into separate files using express.Router() instead of putting everything in one server.js.

Route parameters are a way to make one route handle many different URLs, instead of writing a separate route for each one.

Normally, a route path is fixed, like /categories — it only matches that exact URL.

});

- If someone visits /pages/categories → req.params.name would be "categories"
- If someone visits /pages/projects → req.params.name would be "projects"
- If someone visits /pages/organizations → req.params.name would be "organizations"

Express automatically captures whatever text is in that spot and puts it into an object called req.params, using the name you gave the placeholder (name, in this case). That's why you access it as req.params.name.

So instead of writing four almost-identical route blocks like you have now, you could write one route that handles all of them, using the value from the URL to decide what to render.

AI COACH: No problem at all — this is a subtle one that trips a lot of people up.

Here's the difference:

- <%= %> — outputs the value, but escapes HTML first. So if the value contains characters like <, >, or &, they get converted to safe entities (like &lt;) so they display as plain text rather than being interpreted as HTML. This is the safe default for displaying things like titles, user names, or any text — especially if it comes from user input, since it helps prevent a security issue called XSS (Cross-Site Scripting).

- <%- %> — outputs the value raw, without escaping. If the value contains HTML tags, they'll actually render as HTML on the page.

That's exactly why your title uses <%= title %> (it's just text, safe to escape) while your include(...) calls use <%- %> — because include() returns actual HTML content (the header/footer templates), and you want that HTML to render as HTML, not show up as literal text like <header>...</header> on the page.
