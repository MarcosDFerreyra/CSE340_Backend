STUDENT:
```javascript
app.use((err, req, res, next) => {
    // Log error details for debugging
    console.error('Error occurred:', err.message);
    console.error('Stack trace:', err.stack);

    // Determine status and template
    const status = err.status || 500;
    const template = status === 404 ? '404' : '500';

    // Prepare data for the template
    const context = {
        title: status === 404 ? 'Page Not Found' : 'Server Error',
        error: err.message,
        stack: err.stack
    };

    // Render the appropriate error template
    res.status(status).render(`errors/${template}`, context);
});
```

--------------------------------------------------------------------------------

COACH (translated from Spanish):
Very good example, Marcos — this is a centralized error-handling middleware
in Express (you can tell by the 4 parameters: err, req, res, next). It logs
the error for debugging, determines the status code (using err.status or
defaulting to 500), chooses the corresponding template (404 or 500), and
renders that view with the error message.

Two things catch my attention: first, that you only have two templates

STUDENT (translated from Spanish):

COACH (translated from Spanish):
Excellent answer, Marcos — you nailed it. Showing the stack trace is very
useful in development (it gives you the exact information about where and
why something failed), but in production it's a risk: you could be handing
any user (including a malicious one) internal details of your code — file
names, server paths, database structure, library versions — information
that could be used to find vulnerabilities. Common practice is to show a
generic, user-friendly message in production, and keep the technical detail
only in server logs (like you're already doing with console.error).
